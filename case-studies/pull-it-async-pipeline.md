## 아키텍처 선택

AI 문제 생성은 사용자의 요청-응답 경로에서 오래 걸리는 작업입니다. Main API와 Worker를 분리하고 RabbitMQ로 연결해, Main은 요청을 빠르게 접수하고 Worker가 생성 작업을 처리하도록 설계했습니다. DB 커밋 후에만 메시지를 발행하고, Worker 완료는 별도 큐를 거쳐 Main이 SSE로 전달하도록 했습니다. 생성 성공 상태와 완료 알림 발행 상태도 분리해 알림 실패가 생성 결과를 되돌리지 않게 했습니다.

## 저장 경로 측정

실제 MariaDB와 Hibernate Statistics로 문제 30개 저장 경로를 비교했습니다.

| 방식 | DB SQL 수 | 트랜잭션 수 | 저장 시간 |
| --- | ---: | ---: | ---: |
| Before: 문제별 저장 | 120 | 30 | 약 294–386ms |
| After: 단일 트랜잭션 일괄 저장 | 62 | 1 | 약 90–117ms |

SQL과 트랜잭션 수는 동일 조건에서 결정적으로 확인된 값이며, 시간은 로컬 상황에 따라 변하므로 약 69% 감소라는 상대 변화로 해석합니다.

## 추가 확인

ID 생성 방식을 동일 옵션에서 비교했을 때 `IDENTITY`는 INSERT마다 키를 받아야 하므로 30개 SQL이 필요했고, `SEQUENCE`는 2개 SQL로 배치가 가능했습니다. 다만 전체 스키마의 ID 정책 변경은 위험 범위가 커서, 이번 개선에서는 위험이 낮고 효과가 큰 단일 트랜잭션 일괄 저장을 적용했습니다.

## 해석 범위

- 비교 대상은 실제 MariaDB의 30문제 저장 경로입니다. 대규모 동시 사용자 부하나 전체 서비스 지연을 측정한 값은 아닙니다.
- 공개 저장소에서 아키텍처와 구현을 확인할 수 있으며, 시간 수치는 실행 환경에 따라 달라질 수 있습니다.
